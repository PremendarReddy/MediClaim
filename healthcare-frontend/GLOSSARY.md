# MediClaim - Technical Glossary & Concepts

Complete reference for all technical terms, concepts, and terminology used in the MediClaim platform.

---

## 📚 Core Concepts

### Claim
A request for reimbursement of medical expenses from an insurance company. 

**Fields:**
- `id` - Unique claim identifier (CLM001, CLM002, etc.)
- `patientId` - Associated patient
- `hospitalId` - Hospital that submitted
- `amount` - Claim amount in rupees
- `status` - Current processing stage
- `documents` - Attached supporting docs
- `insuranceId` - Insurance company handling
- `timeline` - Status history with timestamps

**Statuses:**
1. **Pending** - Awaiting hospital documents
2. **Pre-Auth** - Insurance pre-authorization in progress
3. **Approved** - Insurance approved the claim
4. **Rejected** - Insurance denied the claim
5. **Released** - Payment transferred to hospital

---

### Patient
A person seeking medical treatment and insurance claim reimbursement.

**Fields:**
- `id` - Unique patient ID (PAT001, etc.)
- `name`, `email`, `phone` - Contact info
- `dob` - Date of birth (YYYY-MM-DD)
- `aadhar` - Government ID number
- `gender` - Male/Female/Other
- `address` - Residential address
- `status` - Lifecycle status
- `insurance` - Insurance policy details
- `documents` - Uploaded documents
- `createdAt` - Registration date

**Statuses:**
- **Pending** - Awaiting account activation
- **Active** - Full account access
- **Discharged** - Treatment completed, archived

---

### Hospital
A medical facility that submits patient claims to insurance companies.

**Fields:**
- `id` - Unique hospital ID (HOS001, etc.)
- `name` - Hospital name
- `licenseNo` - Government license
- `phone`, `email` - Contact info
- `address` - Physical location
- `bankDetails` - Payment account info
- `createdAt` - Registration date

**Responsibilities:**
- Patient onboarding via OTP
- Report upload and management
- Claim submission to insurance
- Doctor slot management
- Communication with patients

---

### Insurance Company
An organization handling medical claim verification and reimbursement.

**Fields:**
- `id` - Unique insurance ID (INS001, etc.)
- `name` - Company name
- `licenseNo` - Regulatory license
- `phone`, `email` - Contact info
- `totalClaims` - Claims processed
- `approvalRate` - % claims approved

**Responsibilities:**
- Claim review and verification
- Patient document verification
- Coverage evaluation
- Status updates
- Payment processing

---

### Document
Any file uploaded supporting a claim (ID proof, bills, prescriptions, etc.)

**Fields:**
- `id` - Unique document ID (DOC001, etc.)
- `type` - Category of document
- `fileName` - Original filename
- `fileSize` - Size in bytes
- `uploadedBy` - Hospital or Patient
- `claimId` - Associated claim
- `status` - Verification status
- `uploadedAt` - Upload timestamp
- `verifiedAt` - Verification timestamp

**Document Types (8):**
1. `ID_PROOF` - Aadhar, Passport, DL
2. `INSURANCE_CARD` - Policy card copy
3. `PRESCRIPTIONS` - Doctor prescriptions
4. `HOSPITAL_DOCS` - Discharge summary, admission note
5. `BILLS` - Medical bills/invoices
6. `INVESTIGATIONS` - Lab/imaging reports
7. `AMBULANCE` - Ambulance service receipt
8. `BANK_DETAILS` - Patient bank account info

---

### Medical Report
Clinical records generated by hospital about patient's treatment.

**Fields:**
- `id` - Unique report ID (RPT001, etc.)
- `patientId` - Associated patient
- `type` - Type of report
- `fileName` - Report filename
- `doctorName` - Attending doctor
- `department` - Medical department
- `uploadedAt` - Date uploaded
- `remarks` - Additional notes
- `status` - Report status (Normal/Abnormal/Pending)

**Report Types (7):**
1. `BLOOD_TEST` - CBC, blood sugar, etc.
2. `X_RAY` - Chest, limb, spine X-rays
3. `CT_SCAN` - CT scan reports
4. `ULTRASOUND` - Ultrasound imaging
5. `ECG` - Electrocardiogram
6. `DISCHARGE_SUMMARY` - Hospital discharge document
7. `PRESCRIPTION` - Medication prescriptions

---

### Coverage
Insurance policy coverage details and usage tracking.

**Fields:**
- `policyId` - Insurance policy ID
- `totalLimit` - Maximum coverage amount
- `usedAmount` - Amount already claimed
- `remainingCoverage` - Limit - used amount
- `treatments` - Array of covered treatments

**Coverage Breakdown:**
Each treatment has:
- `type` - Treatment name
- `limit` - Maximum coverage for this treatment
- `used` - Already used amount
- `covered` - Boolean (is this treatment covered)

**Covered Treatments (8):**
1. Hospitalization
2. Surgery
3. Medication
4. Diagnostics
5. Doctor Consultation
6. Ambulance
7. Physiotherapy
8. Dental

---

### Doctor Slot
Available appointment time with a hospital doctor.

**Fields:**
- `id` - Slot ID (SLT001, etc.)
- `doctorName` - Doctor's name
- `specialty` - Medical specialty
- `date` - Appointment date
- `time` - Appointment time
- `totalSlots` - Total slots available
- `bookedSlots` - Currently booked count
- `status` - Slot status (Available/Full)

---

### Verification Request
Request from insurance to verify patient information before claim approval.

**Fields:**
- `id` - Request ID (VER001, etc.)
- `claimId` - Associated claim
- `patientId` - Patient to verify
- `requestedInfo` - What needs verification
- `status` - Status (Pending/Verified/Failed)
- `sentAt` - When OTP sent
- `verifiedAt` - When verified
- `notes` - Verification notes

---

### AI Analysis
Results from AI analysis of medical reports or claim data.

**Fields:**
- `id` - Analysis ID (ANA001, etc.)
- `patientId` - Patient analyzed
- `type` - Type of analysis
- `summary` - AI-generated summary
- `metrics` - Extracted key metrics
- `riskLevel` - Low/Medium/High
- `recommendations` - AI recommendations
- `analyzedAt` - Timestamp

---

### Ticket
Support request or communication between users.

**Fields:**
- `id` - Ticket ID (TKT001, etc.)
- `subject` - Ticket subject
- `message` - Initial message
- `status` - Open/Resolved
- `priority` - Low/Medium/High
- `createdBy` - Creator's role
- `createdAt` - Creation timestamp
- `messages` - Message thread

## 🔄 Workflows

### Claim Processing Workflow
```
Hospital Submits Claim
    ↓
Insurance Receives Request
    ↓
Review Documents
    ↓
Verify Patient
    ↓
Evaluate Coverage
    ↓
Approve/Reject Decision
    ↓
If Approved: Process Payment
    ↓
Patient Receives Reimbursement
```

### Patient Onboarding Workflow
```
Hospital Selects "Add Patient"
    ↓
Enter Patient Details
    ↓
System Sends OTP to Email
    ↓
Patient Receives OTP
    ↓
Patient Enters OTP (Demo: 123456)
    ↓
System Verifies OTP
    ↓
Account Created
    ↓
Credentials Sent to Patient
```

### Document Upload Workflow
```
Patient Goes to "My Claims"
    ↓
Clicks "Upload Documents"
    ↓
Selects Document Type
    ↓
Chooses File from Device
    ↓
System Validates File
    ↓
Document Uploaded to Claim
    ↓
Hospital Reviews Document
    ↓
Insurance Verifies Document
    ↓
Status Updated (Verified/Failed)
```

### Verification Workflow
```
Insurance Receives Claim
    ↓
Reviews Documents
    ↓
Needs Patient Verification
    ↓
Creates Verification Request
    ↓
Sends OTP to Patient
    ↓
Patient Enters OTP
    ↓
Patient Verified
    ↓
Insurance Can Approve Claim
```

## 🏗️ Architecture Terms

### Component
A reusable React functional component that renders UI.

**Types in MediClaim:**
- **Page Components** - Full page views (Dashboard, Claims, etc.)
- **Layout Components** - Layout wrappers (DashboardLayout)
- **Feature Components** - Feature-specific UI (DataTable, Sidebar)
- **Card Components** - Reusable card UIs (StatCard, StatusBadge)

### Context
Global state management container using React Context API.

**MediClaim Context:**
- `ClaimContext.jsx` - Contains all global state and functions
- `useClaim()` - Hook to access context from any component

**State Variables:**
- User state (current user, role)
- UI state (dark mode, notifications)
- Data state (claims, patients, documents, etc.)

### Hook
A JavaScript function that lets components use React features.

**Built-in Hooks Used:**
- `useState` - Manage component local state
- `useContext` - Access global context
- `useEffect` - Side effects and lifecycle
- `useNavigate` - Navigate between routes
- `useMemo` - Memoize expensive calculations

### Route
A URL path that displays a specific page component.

**Route Format:**
```
/role/feature/action
/hospital/patients/123
/patient/claims/view
/insurance/claims/doc-review
```

### LocalStorage
Browser's built-in data persistence mechanism (5-10MB limit).

**MediClaim Keys:**
- `user` - Current logged-in user
- `claims` - All claims data
- `patients` - All patients data
- `darkMode` - Theme preference
- etc.

## 🎨 UI Component Terminology

### Badge
Small label showing status or category.

**Types:**
- **StatusBadge** - Shows claim/ticket status (Green/Yellow/Red)
- **RiskBadge** - Shows risk level (Low/Medium/High)

### Card
Container for grouped information.

**Types:**
- **StatCard** - Displays metric with label and trend
- **ClaimCard** - Shows claim summary
- **PatientCard** - Shows patient info

### Modal
Dialog overlay blocking background interaction.

**Uses:**
- Upload documents form
- Confirmation dialogs
- Error messages

### Table
Structured data display in rows and columns.

**Features:**
- Sortable columns (by clicking header)
- Filterable data
- Pagination (if many rows)
- Selectable rows

### Tab
Navigation element switching between content sections.

**Examples:**
- Insurance page: Coverage | Details | Documents
- AI Analysis: Chat | Analysis | History
- Patient Detail: Information | Reports | Claims

## 📊 Data & Statistics Terms

### KPI (Key Performance Indicator)
Measurable metric showing business performance.

**MediClaim KPIs:**
- Total claims submitted
- Approval rate
- Average processing time
- Patient satisfaction
- Claims pending
- Revenue processed

### Risk Level
Assessment of claim fraud/error likelihood.

**Levels:**
- **Low** (Green) - Straightforward claim, high confidence
- **Medium** (Yellow) - Some documentation gaps, moderate risk
- **High** (Red) - Major discrepancies, high fraud risk

### Timeline
Chronological record of status changes.

**Example:**
```
Jan 15 - Claim Submitted
Jan 16 - Pre-Auth Started
Jan 18 - Documents Requested
Jan 22 - Documents Verified
Jan 25 - Claim Approved
Jan 28 - Payment Transferred
```

### Metrics
Extracted key numerical data from documents.

**Examples:**
- Hemoglobin count from blood test
- Dosage from prescriptions
- Cost from bills

## 🔐 Security & Privacy Terms

### OTP (One-Time Password)
Single-use password for account verification.

**MediClaim:**
- Demo OTP: `123456`
- Sent via email during patient registration
- Must be entered to complete onboarding
- Prevents unauthorized access

### Authentication
Process of verifying user identity.

**Flow:**
1. User enters email and password
2. Selects role (Hospital/Patient/Insurance)
3. System verifies credentials
4. Session created and stored in localStorage
5. User can access role-specific pages

### Authorization
Controlling what authenticated user can access.

**Roles:**
- **Hospital** - Can add patients, submit claims, track reports
- **Patient** - Can view claims, upload docs, chat with AI
- **Insurance** - Can review claims, verify documents

### Session
Active logged-in connection for a user.

**Stored in:**
- Browser localStorage under key `user`
- Contains: user ID, email, role, name
- Persists until logout or manual clear

## 🔄 Process Terms

### Pre-Authorization (Pre-Auth)
Insurance approval before treatment/claim submission.

**Purpose:** Ensure procedure is covered before patient pays.

**Status in MediClaim:** `Pre-Auth` (processing stage)

### Reimbursement
Insurance paying back the patient/hospital for medical expenses.

**Flow:**
1. Hospital submits claim
2. Insurance approves
3. Insurance transfers money
4. Patient/Hospital receives payment

### Verification
Insurance confirming information accuracy.

**What's Verified:**
- Patient existence and policy validity
- Document authenticity
- Service eligibility
- Amount accuracy

### Approval Rating
Percentage of claims approved by insurance.

**Example:** 85% approval means 85 out of 100 claims approved.

### Responsive Design
UI that adapts to different screen sizes.

**Breakpoints:**
- Mobile: < 640px
- Tablet: 640px - 1024px
- Desktop: > 1024px

### Dark Mode
Alternative color scheme with dark backgrounds.

**Toggle:** Navbar button switches dark/light mode
**Storage:** Preference saved in localStorage
**Applies to:** All pages automatically

### Loading State
UI indication that data is being fetched/processed.

**Visual:** Spinner, skeleton loader, or disabled button

### Error State
UI showing that something went wrong.

**Display:** Error message, red highlighting, retry button

### Empty State
UI shown when no data available.

**Display:** Icon, message, and call-to-action

## 💡 Best Practices & Principles

### Single Source of Truth
All state stored in one place (ClaimContext).

**Benefit:** Prevents data inconsistency

### Component Reusability
Components designed to be used multiple times.

**Examples:**
- StatusBadge used across all pages
- StatCard used in all dashboards
- DataTable used for all data displays

### Responsive First
Design works on mobile then enhances for larger screens.

**Uses:** Flexbox, CSS Grid, media queries

### Accessibility
Design usable by everyone including disabled users.

**Principles:**
- Keyboard navigation
- Color contrast
- Form labels
- Alt text for images
- Clear error messages

### User Feedback
Immediate response to user actions.

**Implementations:**
- Success notifications
- Error messages
- Loading spinners
- Form validation
- Status updates

## 🔧 Troubleshooting Terminology

### Bug
Unintended behavior or malfunction.

**Examples:**
- Button not working
- Data not saving
- Wrong calculation
- UI not displaying

### Error
Exceptional condition preventing execution.

**Types:**
- JavaScript error (console.error)
- Network error (failed API call)
- Validation error (invalid input)
- Runtime error (undefined variable)

### Console
Browser development tool showing errors and logs.

**Access:** Right-click → Inspect → Console tab

### DevTools
Browser developer tools for debugging.

**Uses:**
- Check console errors
- View localStorage
- Inspect HTML elements
- Monitor network requests
- Debug JavaScript

### Cache
Stored temporary data for faster access.

**Types:**
- Browser cache (images, CSS)
- LocalStorage cache (app data)
